# Act While Thinking: Accelerating LLM Agents via Pattern-Aware Speculative Tool Execution

## 原始摘要

LLM-powered agents are emerging as a dominant paradigm for autonomous task solving. Unlike standard inference workloads, agents operate in a strictly serial "LLM-tool" loop, where the LLM must wait for external tool execution at every step. This execution model introduces severe latency bottlenecks. To address this problem, we propose PASTE, a Pattern-Aware Speculative Tool Execution method designed to hide tool latency through speculation. PASTE is based on the insight that although agent requests are semantically diverse, they exhibit stable application level control flows (recurring tool-call sequences) and predictable data dependencies (parameter passing between tools). By exploiting these properties, PASTE improves agent serving performance through speculative tool execution. Experimental results against state of the art baselines show that PASTE reduces average task completion time by 48.5% and improves tool execution throughput by 1.8x.

## Q&A 论文解读

### Q1: 这篇论文试图解决什么问题？

这篇论文旨在解决大语言模型（LLM）智能体在自主执行任务时，因严格的串行“LLM-工具”循环而产生的严重延迟瓶颈问题。研究背景是，随着LLM推理和理解能力不断提升，基于LLM的智能体已成为实现复杂任务自动化的重要范式。智能体通过将任务分解为子步骤，并交替进行LLM推理和调用外部工具来与环境交互。然而，现有的执行模型强制LLM在每一步都必须等待外部工具执行完成，这种串行依赖导致工具执行时间占总请求时间的35%至61%，成为端到端延迟的主要来源，严重阻碍了智能体服务系统的快速部署和演进。

现有方法的不足在于，它们要么仅关注工具启动或执行环境（如Docker、Conda）的一次性预热优化，这对总执行时间影响甚微；要么专注于通用LLM服务系统优化，但无法有效减少工具执行时间这一核心延迟瓶颈。

因此，本文要解决的核心问题是：如何通过推测性工具执行来隐藏工具延迟，从而加速LLM智能体的整体服务性能。具体而言，论文提出了PASTE方法，其核心洞察是，尽管智能体请求在语义上多样，但在应用层面表现出稳定的控制流（重复出现的工具调用序列）和可预测的数据依赖（工具间的参数传递）。PASTE利用这些模式，将动态的智能体行为抽象为稳定的模式元组，并设计了一个风险感知的调度器，在保证不影响权威（实际）工具执行的前提下，利用空闲计算资源对未来的工具调用进行推测性预执行，从而将工具处理与LLM生成并行化，显著降低任务完成时间。

### Q2: 有哪些相关研究？

本文的相关研究主要可分为三类：系统优化类、推测执行类以及智能体框架与评测类。

在**系统优化类**工作中，已有研究如服务器无服务器工作流和微服务优化通常依赖静态的执行图（如DAG）进行调度和缓存优化。然而，这些方法假设控制流预先可知，不适用于智能体工作负载，因为智能体的工具调用序列是在线动态生成的。本文的PASTE方法则无需预先知晓完整DAG，而是通过识别应用层级的稳定模式（如重复的工具调用序列）来进行推测，从而突破了静态图的限制。

在**推测执行类**工作中，传统推测技术（如CPU分支预测）针对的是有限且结构化的指令集，而智能体请求语义多样、非结构化，直接预测下一个工具调用极其困难。此外，仅预测工具类型不足，还需准确预测其具体参数，而参数常依赖上下文动态生成。PASTE通过分析工具间数据依赖（如参数传递模式），能够有效预测候选参数，从而支持完整的推测工具执行，避免了因参数预测失败或副作用导致的回滚开销。

在**智能体框架与评测类**工作中，现有框架（如ReAct）强制LLM与工具串行执行，导致工具延迟成为瓶颈。相关基准测试（如SWE-bench、Deep Research Bench）揭示了工具执行在总延迟中占显著比例。PASTE与此类工作的关系在于，它基于对这些基准的深入分析，发现了稳定的控制流模式（如编码中的“编辑-验证”模式、研究中的“搜索-访问”模式）和数据流模式，从而设计出能够隐藏延迟的推测执行机制，在保持语义正确性的同时，打破了严格的串行循环。

### Q3: 论文如何解决这个问题？

论文提出的PASTE方法通过**模式感知的推测性工具执行**来解决LLM代理中串行“LLM-工具”循环导致的严重延迟瓶颈。其核心思想是利用代理请求在应用层表现出的稳定控制流（重复的工具调用序列）和可预测的数据依赖（工具间的参数传递），在资源空闲时提前推测执行未来可能需要的工具调用，从而隐藏工具执行延迟。

整体框架包含两个核心组件：**模式抽象**和**机会性推测调度器**。模式抽象负责从历史执行轨迹中挖掘并形式化代理行为的规律，而调度器则负责安全、高效地管理推测性执行与权威性执行之间的资源分配。

**主要模块与关键技术如下：**

1.  **模式抽象与模式元组**：这是PASTE的核心创新。系统定义了**模式元组** $(C, T, f, p)$ 来形式化代理的依赖关系。
    *   **上下文 $C$**：作为控制流锚点，它是一个由工具类型和执行状态（成功/失败）组成的事件签名子序列，剔除了具体查询内容等高方差载荷，从而能在不同语义请求中识别出稳定的控制流模式。
    *   **目标 $T$**：预测未来将要调用的工具类型。
    *   **值映射函数 $f$**：这是关键创新点，用于解决数据流依赖。它不是一个具体的参数值预测（易产生幻觉），而是一个**符号函数**，指定如何从历史事件$C$的载荷中动态推导出目标工具$T$所需的参数（例如，`arg0 = SearchRes["list"][0]["url"]`）。这实现了参数的**延迟绑定**，确保推测执行时能获得精确的参数。
    *   **概率 $p$**：基于历史轨迹验证得出的模式置信度，为调度器的风险感知决策提供依据。
    *   **两阶段挖掘策略**：离线阶段，系统首先基于事件签名挖掘频繁出现的控制流序列（结构规律），然后为这些序列推断对应的值映射函数$f$（数据流逻辑），最后通过验证计算置信度$p$，形成模式池。

2.  **运行时模式预测**：在线运行时，系统持续监控代理执行产生的事件流。当当前事件流的后缀与模式池中某个模式的上下文$C$匹配时，即触发预测。系统会尝试评估该模式对应的值映射函数$f$，利用已产生的具体数据生成一个带有置信度$p$的推测性工具调用事件，并将其提交给调度器。

3.  **机会性推测调度**：调度器是保障系统安全与效率的关键。它严格区分**权威性调用**（由LLM实际生成，必须正确）和**推测性调用**（由模式预测生成，可能出错），并遵循以下设计原则：
    *   **严格优先级与隔离**：权威性任务拥有严格优先级和抢占能力。调度器首先确保满足所有待处理权威性任务的资源需求，必要时会抢占（中止）推测性任务。
    *   **动态闲散资源预算**：推测性任务只能在未被权威性任务占用的“闲散资源”以及一个预设的推测资源预算$B$内执行。
    *   **提升机制**：这是另一个关键创新。当一个权威性请求到达时，如果发现一个正在运行或已完成的推测性任务与之匹配（工具类型和参数一致），则该推测性任务会被立即“提升”为权威性任务。其结果直接提交给代理历史，从而将成功的推测转化为确定的延迟减少，避免了重复计算。
    *   **基于效用的贪婪调度**：对于推测性任务，调度器根据其期望效用（$U = (p \cdot 潜在延迟收益) / (资源成本 \cdot 执行时间)$）进行排序，并贪婪地利用闲散资源执行高效用任务。

4.  **推测资格策略**：为确保安全性和可控性，PASTE引入了用户可配置的策略，用于约束推测行为。策略可以规定哪些工具允许被推测、允许的推测级别（如完全执行、仅预热、干运行等），以及如何处理具有副作用的操作。这使系统操作员能够在性能收益和资源风险之间进行权衡。

总之，PASTE通过创新的模式元组抽象捕获并解耦了代理的控制流和数据流，再通过一个保障安全隔离和具备提升机制的机会性调度器，将不确定的推测转化为确定的性能提升，从而显著加速了LLM代理的任务执行。

### Q4: 论文做了哪些实验？

论文实验主要评估了PASTE方法在加速LLM智能体任务执行方面的性能。实验设置上，PASTE作为一个中间件层实现，部署在LLM推理引擎与外部工具之间，通过推测性执行来隐藏工具调用延迟。

在数据集与基准测试方面，研究采用了包含多种复杂任务的智能体工作负载，这些任务涉及多步骤工具调用（如网络搜索、代码执行、数据查询等）。实验对比了PASTE与现有的先进基线方法，包括标准的串行“LLM-工具”循环执行模式以及其他推测执行策略。

主要结果通过两个关键数据指标体现：平均任务完成时间和工具执行吞吐量。实验结果表明，PASTE能够将平均任务完成时间降低48.5%，同时将工具执行吞吐量提升1.8倍。这验证了通过利用应用层控制流模式（重复出现的工具调用序列）和数据依赖（工具间参数传递）进行模式感知的推测工具执行，能有效打破串行执行的瓶颈，显著提升智能体服务性能。

### Q5: 有什么可以进一步探索的点？

该论文提出的PASTE方法虽然有效，但其核心依赖于对历史任务中稳定控制流（重复的工具调用序列）和可预测数据依赖的识别与利用。这带来了几个明显的局限性和值得探索的方向：

首先，**方法的泛化能力受限**。PASTE在具有固定模式的任务（如数据ETL流程、常规API调用链）上表现优异，但对于高度创造性、探索性或依赖复杂条件分支的任务，其模式可能不稳定，推测执行的准确率会下降，甚至可能因错误推测而增加额外开销。未来研究可以探索如何动态识别和适应**半结构化或演化中的任务模式**，例如引入在线学习机制，实时更新模式库并评估推测置信度。

其次，**推测的深度与风险平衡**。当前工作主要聚焦于加速单个“LLM-工具”步骤的延迟。一个更激进的探索方向是**多步长推测执行**，即基于识别出的模式，一次性推测执行后续多个工具调用。这能进一步隐藏延迟，但参数传递错误和外部状态变更的风险会指数级增加。需要研究更健壮的**推测验证与回滚机制**，例如通过轻量化的形式化方法或运行时检查点来确保系统一致性。

最后，**与LLM推理优化的更深层次结合**。PASTE主要从系统调度层面优化，未来工作可以考虑**将模式知识注入LLM本身**。例如，在提示工程或微调阶段，让LLM显式地学习并输出符合常见模式的、易于推测的结构化计划，从而实现算法（LLM的规划能力）与系统（推测执行引擎）的协同设计，从根源上提升代理的决策效率与可预测性。

### Q6: 总结一下论文的主要内容

该论文针对LLM智能体在任务执行中因串行“LLM-工具”循环导致的严重延迟瓶颈问题，提出了一种名为PASTE（模式感知推测性工具执行）的优化方法。其核心贡献在于利用智能体请求中存在的稳定应用层控制流（重复出现的工具调用序列）和可预测的数据依赖（工具间参数传递）模式，在LLM思考的同时并行执行工具调用，从而隐藏工具延迟。该方法通过识别和利用这些模式进行推测性执行，显著提升了智能体服务性能。实验结果表明，PASTE能将平均任务完成时间降低48.5%，并将工具执行吞吐量提高1.8倍，为加速LLM智能体系统提供了一种有效的解决方案。
